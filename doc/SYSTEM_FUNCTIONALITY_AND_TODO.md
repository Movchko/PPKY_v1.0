## 1. Виртуальные устройства

### 1.1 Исполнитель (пускатель / спичка, Igniter)

- **Текущая реализация**: `VDeviceIgniter` в `device_lib`, цикл прожига: Idle → Run → по таймауту 5 с или обрыв/КЗ → Error.
- **Требование**: после завершения цикла прожига — **повторять цикл заново (пока один раз, но это параметр #define надо иметь возможность менять)** (циклический режим).
- **Уточнение**: логика повторения - **если проверка не показывает обрыв (КЗ возможно интерпретируется как норма)**
- **Местоположение**: `device_lib/src/device_igniter.cpp`.

### 1.2 РЕЛЕ (новое виртуальное устройство)

- **Двухканальное** реле: 1 канал — для системы, 2 канал — ноги с процессора для проверки переключения.
- **Обратная связь по GPIO**: подтверждение, что реле переключилось (чтение состояния на втором канале).
- **Местоположение**: добавить `device_relay.hpp/cpp` в `device_lib`, `DeviceRelayConfig` в `device_config.h`, тип `DEVICE_RELAY_TYPE` в `device.hpp`.

### 1.3 ДПТ (датчик превышения температуры)

- **Текущая реализация**: расчёт сопротивления по АЦП, пороги Пожар/Норма/Обрыв/КЗ, режим концевика, термопара MAX31855 в MCU_TC.
- **Требование**:
  1. **Режим определения сопротивления на входах** (по АЦП) — идёт первым.
  2. Если сопротивления **нет** (ниже порога) — переключать измерение температуры на **термопару** (MAX31855 по SPI).
- **Порог «нет сопротивления»**: конфигурируемый (пока). После нахождения нужного значения будет одинаковым.
- **Местоположение**: `device_lib/src/device_dpt.cpp`, `MCU_TC` (MAX31855 уже есть).

### 1.4 Концевик / кнопка (новое виртуальное устройство)

- Конфигурацией выбирается: **концевик** или **кнопка**.
- В конфигурации задаётся **действие по срабатыванию**:
  - открытие или пожар в выбранных зонах — **одна команда на зону**;
  - запуск сценария тушения: **Пуск общий**, **Пуск**, **выбранные зоны**.
- **Местоположение**: добавить `device_limit_switch.hpp/cpp` в `device_lib`.

## 3. Общие команды и протокол (Backend)

### 3.1 Команды ПОЖАР

- **Команда ПОЖАР** и **ответ на ПОЖАР** — сервисные, общие для всех устройств.
- **Инициатор**: любое устройство на шине (ППКУ, МКУ, ДПТ, концевик и т.д.).
- **Таймаут валидации**: 2 с. По посылке команды устройство ждёт ответ (валидацию).
- Если валидация **не пришла** за 2 с — выставить флаг `fire_not_response`.
- **Местоположение**: `device_lib/include/backend.h`, `device_lib/src/backend.c`, добавить `ServiceCmd_Fire`, `ServiceCmd_FireAck`.

### 3.2 Команды задания адресов (кольцо CAN)

- **Топология**: ППКУ CAN1 ↔ МКУ1 CAN1 ↔ МКУ1 CAN2 ↔ МКУ2 CAN1 ↔ … ↔ ППКУ CAN2.
- **Алгоритм**:
  1. ППКУ шлёт команду задания адреса с **параметром в данных** (начинается с 0).
  2. МКУ получает команду: если параметр = 0, значит он первый — задаёт себе адрес 1, увеличивает параметр на 1, пересылает ту же команду в другой CAN с параметром 1.
  3. Следующий МКУ: параметр 1 → адрес 2, параметр 2, пересылает дальше. И так далее.
  4. Последний МКУ шлёт команду обратно ППКУ — кольцо цело, адреса заданы, количество = параметр.
- **Структура адреса** (из `Device`):
  ```c
  typedef struct {
      uint8_t zone;   /* Номер зоны */
      uint8_t l_adr;  /* Младшая часть адреса */
      uint8_t h_adr;  /* Старшая часть адреса — задаётся командой */
      uint8_t d_type; /* Тип устройства */
  } Device;
  ```
- **Местоположение**: backend, добавить `ServiceCmd_SetAddress`, `ServiceCmd_AddressRingComplete`.

### 3.3 Команда «запомнить конфигурацию системы»

- ППКУ должна уметь **сохранять конфигурацию системы** (всех МКУ) для сравнения при следующем старте.
- **Местоположение**: backend, `ServiceCmd_SaveSystemConfig` или аналог.

## 4. Параметры исполнителей и режимы тушения

### 4.1 Задержки исполнителя

- У каждого исполнителя:
  - **Задержка зоны** (1);
  - **Задержка модуля** (2).
- **Местоположение**: `DeviceIgniterConfig` (и аналоги для других исполнителей).

### 4.2 Режимы работы системы тушения

- **Автоматический**, **автономный**, **ручной** — одинаковы для ППКУ и МКУ.
- Задаются в **ППКУ**.
- **Поведение**:
  - **Автоматический**:
    1. МКУ отправила ПОЖАР → ППКУ приняла, ответила МКУ.
    2. ППКУ запускает таймер (например 30 с) перед запуском тушения.
    3. По истечении таймера — ППКУ посылает команду начать тушение.
    4. Таймер можно прервать: нажатием кнопки «тушить» (принудительно) или «остановить».
  - **Автономный**: как автоматический, но если **ППКУ не ответила** на запрос МКУ (fire_not_response) — МКУ сами начинают сценарий по своим задержкам.
  - **Ручной**: задержка зоны, **всегда только после нажатия кнопки**.

## 5. ППКУ: хранение и проверка настроек МКУ

- ППКУ хранит настройки всех МКУ.
- При запуске: считывает настройки (или контрольную сумму) и **сравнивает**.
- Если настройки **отличаются** — сообщить определённым **флагом**.
- Далее — функционал логирования и конфигурирования.

## 6. Силовой ключ питания (BTS740S2)

- **Текущая реализация**: `PControl` в `stm_PPKY`, GPIO ST/OUT, без прерывания по статусу.
- **Сигнал ST (GPIO)**:
  - **ST = 1**: питание выключено, либо КЗ/неисправность.
  - **ST = 0**: питание включено и всё в норме.
- **Требование**:
  1. Настроить **прерывание на ST** (по фронту/уровню при переходе в 1).
  2. При ST = 1 при включённом питании — **сразу отключать** выход.
  3. Пытаться **повторно включить** через N мс (пока 2), до M попыток (пока 1 с).
  4. После неудачи — выставить **флаг ошибки питания**.
- **Местоположение**: `stm_PPKY/Core` (power_control, main, прерывания).

## 8. Подсветка (LED)

### 8.1 Режимы подсветки

- **Сигнальный**: максимальная яркость (вкл/выкл).
- **Дежурный**: одно настраиваемое значение яркости для всех дежурных LED, хранится в ППКУ.

### 8.2 Статусные лампочки

| # | Константа      | Описание                          |
|---|----------------|-----------------------------------|
| 0 | LED_POWER      | Лампочка питания                  |
| 1 | LED_NORM       | НОРМА                             |
| 2 | LED_START      | ПУСК                              |
| 3 | LED_STOP       | ОСТ ПУСК                          |
| 4 | LED_ERR        | Неисправность                     |
| 5 | LED_FIRE       | Пожар                             |
| 6 | LED_AUTO_OFF   | Режим тушения ручной (автоматика выкл.) |

### 8.3 Подсветка кнопок

- **Большие**: LED_BUT_START_ALL (7), LED_BUT_STOP (8), LED_BUT_START_SP (9) — только сигнальный режим.
- **Подписи**: LED_STR_START_ALL (10), LED_STR_STOP (11), LED_STR_START_SP (12).
- **Мелкие**: LED_BUT_ENTER_UP (13), LED_BUT_ESC_DW (14) — при нажатии сигнальный, через 10 с — дежурный.

### 8.4 Специальная логика

- **ПУСК СП**: надпись всегда блёклая. При удержании 3 с — обратный отсчёт на дисплее, надпись в сигнальном, кнопка горит. Пока держим — надпись мигает 0.5 Гц. По истечении 3 с — надпись и кнопка в сигнальном, загорается LED_START.
- **При команде ПОЖАР**: мигает LED_FIRE, бипер в режиме пожара (новый режим).
- **Режимы тушения при ПОЖАР**:
  - Автоматический: все надписи и кнопки ПУСК СП, ОСТАНОВ в сигнальный. По нажатию ПУСК СП — они гаснут, загорается ПУСК Общий. По ответу МКУ — LED_FIRE в дежурный. После ответа МКУ — остановить тушение по кнопке нельзя. После завершения всеми МКУ — ПУСК Общий выключается.
  - Кнопка ОСТАНОВ при пожаре переводит режим из автоматического в ручной.
  - Автономный — надстройка над автоматическим (МКУ без ответа ППКУ сами стартуют).
- **LED_ERR**: при неисправности — сигнальный 3 с, затем дежурный. Новая неисправность — снова сигнальный. На экране — бегущая строка: **код ошибки = имя зоны**, где возникла неисправность.
- **LED_POWER**: два входа питания. Если по одному пропало — мигание 5 с (период 0.2 с), затем дежурный. Неисправность по питанию может пропасть; остальные — только через рестарт.

### 8.5 Текущая реализация

- `led.c` в stm_PPKY: I2C драйвер, Led_Set, Led_SetBrightness, Led_Process (ENTER/ESC: 10 с возврат к дежурному). Нет разделения сигнальный/дежурный для статусных, нет логики пожара и кнопок ПУСК/ОСТАНОВ.
